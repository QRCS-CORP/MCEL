\chapter{MCEL\+: Merkle Chained Event Ledger }
\hypertarget{index}{}\label{index}\index{MCEL: Merkle Chained Event Ledger@{MCEL: Merkle Chained Event Ledger}}
\hypertarget{index_mcel_overview}{}\doxysection{\texorpdfstring{Overview}{Overview}}\label{index_mcel_overview}
MCEL is a Merkle rooted, append-\/only event ledger designed for tamper-\/evident logging and audit. Records are hash-\/committed, blocks aggregate record commitments into a Merkle root, and checkpoints cryptographically bind block roots into a signed chain suitable for long-\/term verification.\hypertarget{index_mcel_design}{}\doxysection{\texorpdfstring{Design Summary}{Design Summary}}\label{index_mcel_design}
The library exposes a storage abstraction through callbacks, allowing deployment over files, databases, object stores, or custom persistence layers. The ledger provides\+:
\begin{DoxyItemize}
\item Deterministic record commitment and append
\item Merkle block sealing and root commitment
\item Signed checkpoint bundling and checkpoint chain verification
\item Inclusion proof generation and verification for audit exhibits
\end{DoxyItemize}\hypertarget{index_mcel_storage}{}\doxysection{\texorpdfstring{Storage Contract}{Storage Contract}}\label{index_mcel_storage}
MCEL uses a caller-\/supplied storage backend via callback functions. The backend must support querying size, reading, writing, and appending objects identified by location keys.\hypertarget{index_mcel_security}{}\doxysection{\texorpdfstring{Security Notes}{Security Notes}}\label{index_mcel_security}
MCEL integrity depends on collision resistance of the hash function and unforgeability of the signature scheme used for checkpoints. Implementations should protect signing keys, enforce strict monotonic sequencing where required by policy, and ensure storage callbacks are atomic for append operations.\hypertarget{index_mcel_modules}{}\doxysection{\texorpdfstring{API Modules}{API Modules}}\label{index_mcel_modules}
